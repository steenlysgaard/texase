* Versioning
- Follow a semantic versioning approach based on https://softwareengineering.stackexchange.com/questions/255190/how-does-semantic-versioning-apply-to-programs-without-api
- MAJOR.MINOR.PATCH
  - Increment the patch number when you ship a bug fix, e.g.: “fixed bug where application always crashes at 00:00 PM on Tuesdays”.
  - Increment the minor version number when adding a new feature, e.g.: “added support for underlined text”.
  - Increment the major version when you significantly overhaul the user interface, or rewrite all internals. E.g.: “Toolbars use too much screen estate. The UI is now provided through touch gestures only” – such a change would break existing workflows.


* Release
When ready for a release:
1. Decide on a version, follow [[Versioning]] above
2. Put the version in pyproject.toml, update uv.lock (by running =uv lock=), and add that as a final commit of that release
3. Go to GitHub and create a new release, give it the same tag as the version
4. A github action takes care of publishing to PyPI
5. Bump the version (add one to the PATCH number) and add a b in the end, signifying beta. Update uv.lock.
6. Commit this to main.
7. Update version on Niflheim by updating the version number in the eb file and injecting new checksums, install the new version and delete the previous.
